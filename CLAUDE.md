# MowgliNext

Open-source autonomous robot mower monorepo. ROS2 Kilted, Nav2, and a GTSAM iSAM2 factor-graph localizer (`fusion_graph`, in `ros2/src/fusion_graph/`) that is the **sole and default** map+odom localizer ‚Äî it owns **both** `map‚Üíodom` AND `odom‚Üíbase_footprint` and adds optional LiDAR scan-matching + loop-closure factors. **The robot_localization dual-EKF (`ekf_map_node` + `ekf_odom_node`), `config/robot_localization.yaml`, slam_toolbox, and Kinematic-ICP were all removed**; `navigation.launch.py` launches `fusion_graph_node` unconditionally (there is no `use_fusion_graph` arg). BehaviorTree.CPP v4, cell-based strip coverage. (`docs/HANDOFF_FUSION_GRAPH.md` is historical ‚Äî the migration record, **not** the steady-state reference; see [`wiki/Architecture.md`](https://github.com/mowglinext/mowglinext/wiki/Architecture#optional-factor-graph-localizer-fusion_graph) for current behaviour.) **Drive base:** the wheel-velocity loop runs in STM32 firmware (vendored PX4 PID); ROS2 sends `cmd_vel` and the firmware closes the per-wheel loop.

**Website:** https://mowgli.garden | **Wiki:** https://github.com/mowglinext/mowglinext/wiki | **First-boot checklist:** [`docs/FIRST_BOOT.md`](docs/FIRST_BOOT.md)

## Safety ‚Äî READ FIRST

This robot has spinning blades. The STM32 firmware is the sole blade safety authority.

- NEVER bypass firmware blade safety checks from ROS2
- Blade commands from ROS2 are fire-and-forget ‚Äî firmware decides whether to execute
- Emergency stop is handled by firmware, not software
- Flag ANY change that could affect physical behavior as safety-critical in PR reviews

## Architecture Invariants (DO NOT VIOLATE)

1. **Localizer = `fusion_graph` (GTSAM iSAM2), sole and default.** `fusion_graph_node` is launched unconditionally by `navigation.launch.py` and publishes **both** `map‚Üíodom` AND `odom‚Üíbase_footprint` (replacing the removed `ekf_map_node` + `ekf_odom_node`). There is **no `use_fusion_graph` launch arg** ‚Äî do not write it from the GUI. It runs a Pose2 graph (`node_period_s` cadence) with a custom `GnssLeverArmFactor` (analytic Jacobian ‚Äî antenna lever-arm rotates with the node's yaw), a wheel between-factor with non-holonomic covariance (œÉ_y << œÉ_x), a gyro between-factor on yaw, and COG/mag yaw unary factors. Inputs it still consumes (unchanged from the old EKF wiring): `/gps/pose_cov` (from `navsat_to_absolute_pose_node`, datum + lever-arm from `mowgli_robot.yaml`), `/imu/cog_heading` (GPS COG yaw, `cog_to_imu_node`), `/imu/mag_yaw` (when calibrated). RTK-Fixed œÉ ~3 mm flows through as-is. **LiDAR scan-matching between-factors and loop-closure factors are wired**, gated by `use_scan_matching` / `use_loop_closure` (the GUI's LiDAR toggle drives these), letting the map-frame estimate ride through multi-minute RTK-Float windows. Surface: `/fusion_graph/diagnostics` (1 Hz `DiagnosticArray`: `total_nodes / loop_closures / scan_matches_ok|fail / cov_xx|yy|yawyaw`), `/fusion_graph/markers`, `/imu/fg_yaw`, services `~/save_graph` + `~/clear_graph` (`std_srvs/Trigger`, in the GUI's Diagnostics ‚Üí Fusion Graph panel).

   **slam_toolbox, Kinematic-ICP, and FusionCore are gone.** The previous RTK-fallback architecture (parallel TF tree, `slam_pose_anchor_node` EWMA, `wheel_odom_tf_node`, `slam_scan_frame_relay`, `slam_map_persist_node`, `rtk_pose_mux_node`, fusion as `pose1`), the K-ICP encoder adapter, and the FusionCore 22D UKF were all removed in favor of the factor-graph approach. `fusion_graph` is the only LiDAR-aware localizer; the `kinematic_icp` and `fusioncore` submodules are no longer part of the build.
2. **TF chain follows REP-105** ‚Äî `map ‚Üí odom ‚Üí base_footprint ‚Üí base_link ‚Üí sensors`. **Both `map‚Üíodom` AND `odom‚Üíbase_footprint` are published by `fusion_graph_node`** (the `ekf_map_node`/`ekf_odom_node` that used to own them are removed). Do NOT publish either TF from any other node. All Nav2 nodes use `base_footprint` as the robot frame. `base_link` is at the rear wheel axis (OpenMower convention, do not move).
3. **Cyclone DDS** ‚Äî not FastRTPS (stale shm issues on ARM)
4. **Map frame = GPS frame** ‚Äî X=east, Y=north, no rotation transform
5. **Costmap obstacles disabled in coverage mode** ‚Äî collision_monitor handles real-time avoidance
6. **dock_pose + tool_width live in `mowgli_robot.yaml`** ‚Äî single source of truth. `dock_pose_x/y/yaw`: `calibrate_imu_yaw_node`'s dock pre-phase and `/map_server_node/set_docking_point` (called by the GUI to manually pin the dock to the current robot pose) both write back to `mowgli_robot.yaml` via line-splice updates that preserve comments. `hardware_bridge`, `map_server`, and `dock_yaw_to_set_pose` read the values as ROS parameters. There is no `dock_calibration.yaml` anymore. `tool_width` (effective blade cut width): `navigation.launch.py` flows the same scalar into `map_server.tool_width` (mark_cells_mowed stamp radius + sliver detection) AND `coverage_server.operation_width` (F2C `Robot::setCovWidth`, controls swath spacing). The two used to be configured separately (`mower_width=0.18` + static `operation_width=0.20`), which left thin un-mowed strips between adjacent F2C swaths because map_server's stamp radius was narrower than F2C's spacing.
7. **Multi-area coverage = explicit segments from `plan_coverage` (F2C v3, no turn planning).** `map_server_node` owns area polygons and the `mow_progress` grid layer. BT outer loop `GetNextUnmowedArea` iterates areas; per area, `PlanCoverageArea` calls `map_server_node/get_mowing_area` (outer ring + obstacle holes) and forwards it to `mowgli_coverage`'s `plan_coverage` action (`mowgli_interfaces/action/PlanCoverage`, Fields2Cover v3 at `/opt/fields2cover-300`). The result carries BOTH an ordered list of discrete straight segments ‚Äî concentric headland rings (outermost first, each split into corner-to-corner arcs) then straight serpentine swaths, kept for the GUI and resume bookkeeping ‚Äî AND one or more continuous, **hole-free** `drivable_subpaths` (`buildContinuousSubPaths`) that join the rings + swaths with **forward turn-around arcs** of nominal radius `connector_turn_radius` (default 0.18 m, from `mowgli_robot.yaml`, floored at `min_turning_radius`) ‚Äî cusp-free, bounded to the chassis-safety inset. A hole-free field yields exactly one sub-path (== the whole `full_path`); a multi-hole field is split so each obstacle gap becomes a blade-off Nav2 transit, and the sub-paths are ordered by a deterministic greedy nearest-neighbour pass to minimise that blade-off transit. `FollowStrip` drives each sub-path as ONE continuous `FollowCoveragePath` goal (FTCController tracks it end-to-end ‚Äî no per-segment dispatch), bridging consecutive sub-paths with a blade-off NavigateToPose transit that routes around the obstacle, and resumes mid-area by trimming at a pose cursor into the sub-path concatenation (plan is deterministic so the cursor is stable across re-plans). `full_path` is the concatenation, kept for the GUI. Too-small areas are rejected by the server (`success=false`), not by a BT-side size gate.
8. **FTCController for coverage; RotationShim+RPP for transit.** The `FollowCoveragePath` slot is `mowgli_nav2_plugins/FTCController` ‚Äî a deterministic Follow-the-Carrot controller with a decoupled lon/lat/ang PID (low-pass on the derivative). **RESTORED 2026-06-19**, reverting an MPPI experiment. FTC does ONE in-place `PRE_ROTATE` pivot to the path-start heading, then tracks the continuous `full_path` tightly (following the forward turn-around arcs), and skirts obstacles by deviating the path LATERALLY in real time (`enable_obstacle_deviation`, reading the LiDAR `obstacle_layer`; off in the no-LiDAR overlay). MPPI was reverted because as a SAMPLING controller it cut corners / made omega-loops at swath U-turns, and every attempt to sharpen its corners made it weave on the straights. The `FollowPath` (transit) slot uses RotationShim wrapping `nav2_regulated_pure_pursuit_controller` (RPP). **Nav2 params are a shared base + thin overlays:** `nav2_params_base.yaml` (everything common) is deep-merged with `nav2_params_lidar.yaml` OR `nav2_params_no_lidar.yaml` (only the genuine differences ‚Äî costmap obstacle vs static layers, scan-based vs pass-through collision_monitor, `FollowPath.use_collision_detection`), selected by the `use_lidar` arg in `navigation.launch.py`. Edit shared params in the **base**; variant params in the **overlay**. `test_nav2_params.py` validates the merged result and pins the variants in lockstep.
9. **Emergency auto-reset on dock** ‚Äî When emergency is active and robot is on dock (charging detected), BT auto-sends `ResetEmergency` to firmware. Firmware is sole safety authority and only clears latch if physical trigger is no longer asserted.
10. **Undock via Nav2 BackUp behavior** ‚Äî BackUp (1.5m, 0.15 m/s) via `behavior_server`, not `opennav_docking` UndockRobot (isDocked() unreliable with GPS drift near the dock). Costmaps are cleared after undock.
11. **Zero-odom only when charging AND idle** ‚Äî `hardware_bridge_node` does not reset odometry during undock sequence.
12. **Battery current for dock detection** ‚Äî Hardware bridge publishes `abs(charging_current)` when charging, `0.0` when not, for `SimpleChargingDock` compatibility.
13. **Docking server cmd_vel** ‚Äî Remapped to `/cmd_vel_docking` through twist_mux (priority 15).
14. **Coverage grid_map ‚Üí OccupancyGrid convention** ‚Äî easy to get wrong. `grid_map::GridMap::getSize()(0)` = cells along X, `getSize()(1)` = cells along Y. grid_map iterates with `r=0 ‚Üí X_max` (decreasing) and `c=0 ‚Üí Y_max` (decreasing). `nav_msgs/OccupancyGrid` has `width = X_cells`, `height = Y_cells`, row-major with `data[y_row * width + x_col]`, where `col=0 ‚Üî origin.x` (X_min) and `row=0 ‚Üî origin.y` (Y_min). To convert a grid_map cell `(r, c)` to an OccupancyGrid index: `og_col = nx - 1 - r`, `og_row = ny - 1 - c`, `flat_idx = og_row * nx + og_col`. ALWAYS set `mask.info.width = nx` and `mask.info.height = ny` ‚Äî swapping produces a 90¬∞-rotated mask, which silently marks valid interior polygon cells as lethal and breaks Smac planning with "Start occupied" errors. See `mow_progress_to_occupancy_grid()` in `map_server_node.cpp` as the reference implementation.
15. **Robot config is a SPARSE installed file over a TEMPLATE of defaults.** The *installed* `mowgli_robot.yaml` (`/ros2_ws/config/mowgli_robot.yaml`, seeded from `install/config/mowgli/mowgli_robot.yaml`) is **sparse** ‚Äî it holds ONLY install-time choices (datum, NTRIP, `lidar_enabled`, GNSS hardware, `mower_model`) + calibration outputs (dock_pose, `ticks_per_meter`, `wheel_pid_*`, `imu_yaw`, mag). **Every other parameter's DEFAULT lives in the in-package template `ros2/src/mowgli_bringup/config/mowgli_robot.yaml`.** At launch, `ros2/src/mowgli_bringup/launch/robot_config_util.py` (`load_robot_params`) **DEEP-MERGES the sparse installed config OVER the template**, so an absent key falls through to its template default and nodes always receive a complete parameter set. Consequences: a maintainer changing a template default propagates to every robot that hasn't explicitly overridden it; the GUI can **reset any setting to its default** (Settings page: an "overridden" dot + a reset button; reset = delete the key so it falls back to the template ‚Äî the GUI backend also prunes any saved-value-equals-default key on write, keeping the file sparse); an older FULL installed config still works (it merges to itself, a no-op). To change a value, set it in the installed file; to restore a default, DELETE its line (do NOT copy the template value in).

## What NOT to Do

- Do NOT add ROS1 patterns (rosserial, roscore, catkin) ‚Äî this is ROS2 only
- Do NOT use FastRTPS ‚Äî Cyclone DDS is required
- Do NOT mock the database/firmware in integration tests ‚Äî use real interfaces
- Do NOT publish a `map‚Üíodom` (or `odom‚Üíbase_footprint`) TF from any node other than `fusion_graph_node`. It is the sole, default, unconditional localizer and owns BOTH transforms ‚Äî there is no `ekf_map_node` fallback and no `use_fusion_graph` launch arg (both were removed).
- Do NOT re-introduce slam_toolbox, Kinematic-ICP, or FusionCore. All three were tried and removed: slam_toolbox required a parallel TF tree + EWMA anchor + gating that was fragile across RTK-Float windows; K-ICP only gave a body-frame twist with no absolute-pose information; FusionCore's 22D UKF would not collapse œÉ_xy below ~30 cm even under RTK-Fixed. The factor-graph design (`fusion_graph`) is the replacement ‚Äî LiDAR enters the same graph as scan-matching between/loop-closure factors (see `wiki/Architecture.md` ¬ß *Optional: Factor-Graph Localizer*).
- Do NOT send blade commands without firmware safety checks
- Do NOT hardcode GPS coordinates, dock poses, or NTRIP credentials
- Do NOT use RPP or MPPI for coverage paths ‚Äî the coverage slot is `mowgli_nav2_plugins/FTCController` fed the continuous `full_path`. MPPI was tried and reverted 2026-06-19 (cut corners / omega-looped at U-turns, weaved on the straights); do NOT switch coverage back to it without re-reading the FTC block + tuning notes in `nav2_params_base.yaml`.
- Do NOT re-introduce F2C turn planning (Dubins/CC-Dubins/Reeds-Shepp) into coverage. Every F2C turn variant was field-tested and failed (forward-only turns omega-loop at 0.16 m spacing; Reeds-Shepp cusps are untrackable ‚Äî FTC follows arcs but `forward_only: true` means it does not reverse mid-swath). The continuous `full_path` joins rings+swaths with CUSTOM forward turn-around arcs (`buildContinuousPath`, cusp-free, bounded to the safety inset) ‚Äî NOT F2C's turn planners, which remain banned.
- Do NOT use `base_link` as robot_base_frame in Nav2/robot_localization ‚Äî use `base_footprint` (REP-105)
- Do NOT use opennav_docking UndockRobot ‚Äî use Nav2 BackUp behavior (isDocked() unreliable with GPS drift)
- Do NOT refer to `mower_width` ‚Äî the param is `tool_width`, lives in `mowgli_robot.yaml`, and is the single source for both `map_server.tool_width` and `coverage_server.operation_width` (injected at launch). Reintroducing a separate `mower_width` will re-open the swath-spacing-vs-stamp-radius gap that caused 54 % coverage in 2026-05-12.
- Do NOT copy pure defaults into the *installed* `mowgli_robot.yaml` (see Invariant 15). Defaults belong in the in-package template `mowgli_bringup/config/mowgli_robot.yaml`; the installed file is SPARSE and holds only install/site choices + calibration outputs + genuine overrides. Padding it with default values defeats "reset to default" and the propagate-a-new-default-to-all-robots property. To add a NEW parameter's default, add it to the template ‚Äî not the sparse installed file.
- Do NOT use `StoppedGoalChecker` for the `coverage_goal_checker` slot ‚Äî use `mowgli_nav2_plugins/PathProgressGoalChecker`. StoppedGoalChecker fires on velocity stoppage and matches FTC's PRE_ROTATE pivots mid-traversal, completing the coverage action at <2 % coverage.
- Do NOT use the upstream `opennav_coverage` server. The repo carries it as a git submodule for the `opennav_coverage_msgs` action definitions only ‚Äî every other subpackage (`opennav_coverage`, `_bt`, `_demo`, `_navigator`, `opennav_row_coverage`) is `COLCON_IGNORE`'d. The upstream server is pinned to F2C 1.2.1 (no `TrapezoidalDecomp`, no `generateHeadlandSwaths`, no `planPathForConnection`). Use `mowgli_coverage` (F2C 2.0.0).
- Do NOT call `buildHeadlandRingPath()` from the BT side ‚Äî the coverage server owns headland traversal (`generateHeadlandSwaths()` ring segments in `planBoustrophedon`).
- Do NOT reintroduce the hardcoded sim test polygon into `ros2/src/mowgli_map/config/map_server.yaml`. A non-empty default makes a fresh real-robot install try to mow a phantom polygon at the GPS datum the moment `COMMAND_START` fires. The sim injects its polygon via `sim_full_system.launch.py` parameter override.
- Do NOT add a `locks:` block to `twist_mux.yaml` without first wiring a publisher in `hardware_bridge_node`. The earlier `/emergency_stop` lock was listening on a topic no node ever published; the real e-stop path is the `/hardware_bridge/emergency_stop` service which sets the firmware latch.
- Do NOT build a GPS wrong-fix gate around an **unbounded "since last accepted fix" motion accumulator**. The `GnssMobileGate` (erroneously bundled into PR #307, reverted from dev + feat/ftc-revive) compared each fix to `expected_motion = max(wheel,cmd)-delta since the last ACCEPTED fix`, which only reset on acceptance ‚Äî so once it started rejecting, `expected_motion` ran away (observed 20.9 m for <1 m real motion), the innovation stayed huge, **every fix rejected forever**, GPS locked out, and `cov_xx` ballooned to ~2.5 m œÉ. A motion-consistent gate MUST compare the GPS step to *bounded per-interval* travel (the live `fusion_graph` gate: `jump > rtk_wrongfix_max_jump_m + lever_sweep + wheel_dist`). Signature: `gps_gate_decision=rejected` + `gps_rejects_wrongfix` climbing + `expected_motion` >> real motion + œÉ growing.
- Do NOT leave `controller_server.odom_topic` unset ‚Äî Nav2 defaults it to `"odom"`, which **nothing publishes** on this robot, so MPPI/RPP/FTC get ZERO velocity feedback (MPPI circles on tight coverage; RPP's velocity-scaled lookahead collapses ‚Üí transit weaving). Point it at a published source: `/wheel_odom` (carries vx + angular.z) is the live choice. `/odometry/filtered` is only usable if fusion_graph is publishing its `twist.angular.z` (it leaves it 0 by default). `test_nav2_params.py` guards this.
